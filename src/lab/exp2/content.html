Systems and Software Virtual Lab
Process Control
Introduction

Welcome to Process Control, the second experiment in the Systems and Software Virtual Lab. Process control refers to creating, killing, changing priorities of processes on a machine. The goal of this experiment is to teach basic process control on a Linux machine. The theory section provides a brief overview of commands needed to do process control.

The experimental platform for this experiment is a command line interface which mimics the terminal of a linux machine. The command line interface supports few basic set of commands to enable the user learn process control basics. The experiment contains the command line interface and an instructor section (in green background) below it. The instructor section guides the user to peform the experiment which is a series of tasks.

Proceed to the theory section to get an overview of the commands and then move to the experiment section to get some hands-on experience and finish the experiment tasks. After you are done with the experiment go to the quiz section to test your understanding.

Theory

1. Viewing the processes (the ps command):

ps command is the short form of process status. ps displays information about a selection of the active processes when it is executed without any flags. 


johndoe@laptop:~/Documents$ ps
   PID   TTY        TIME    CMD
  2635   pts/9    00:00:00  ps
  26439  pts/9    00:00:00  bash

The most used flags for ls command are A, e, p, C.

FlagPurpose
ADisplays all the processes running on a system
eSame as 'A' option
pDisplays processes whose process numbers occur in pidlist (ps -p pidlist)
CDisplays processes corresponding to a particular command 

johndoe@laptop:~/Documents$ ps -e
PID  TTY          TIME CMD
1    ?        00:00:00 init
2    ?        00:00:00 kthreadd
3    ?        00:00:00 migration/0

3111 pts/0    00:00:00 bash

The above is the abridged version of ps -e 


johndoe@laptop:~/Documents$ ps -p 3111 3151

2. Running processes in the background

When we use the shell to launch a process, we wait for the process to be completed before we get back the control of the shell.This is called running a process in the foreground. It is also possible to run a process in the background. In this case, the program is launched and control is returned the the shell immediately, allowing you to run other commands. In the meantime, the initial process continues to run in the background. To place a process in the background, we append '&' at the end of the command. 


johndoe@laptop:~/Documents/my_projects$ gedit filename.txt &
johndoe@laptop:~/Documents/my_projects$

We can use the 'jobs' command to view the background processes.

johndoe@laptop:~/Documents/my_project$ jobs
[2]+  Done                    gedit filename.txt

If a process running in the foreground is taking a long time, we can relegate it to the background. This is a 2 step process.
a). Hold Ctrl + Z key to suspend the process.
Use jobs to see the current status of the process.

johndoe@laptop:~/Documents/my_project$ jobs
[2]  + Suspended        gedit filename.txt

b) Now type in 'bg %N' where N is the job id - 2 in this case. 

johndoe@laptop:~/Documents/my_project$ bg %2
[2] gedit filename.txt &

Now control will return to the shell and the process will execute in the background.
To bring back the process to the foreground, we use the 'fg' command.

johndoe@laptop:~/Documents/my_project$ fg %2

Now the shell prompt goes away and control is given back only after process completes the execution.

3. Killing processes

We use the kill command to kill the processes.The kill command takes the PID of the process as an argument.
Suppose we want to kill the process 'gedit' 

johndoe@laptop:~/Documents$ ps -e | grep gedit
3155 ?        00:00:05 gedit

The first column of the output gives the PID of the gedit process 

johndoe@laptop:~/Documents$ kill 3155 
johndoe@laptop:~/Documents$

We can now check whether the process has been killed or not.

johndoe@laptop:~/Documents$ ps -e | grep gedit
johndoe@laptop:~/Documents$

Since there is no output, it means that there is no process running with the name gedit, which means that it has been killed.

4. Changing Process Priority:

The priority of a process is indicated by its "nice value". Lesser the nice value of a process, more is the priority. Nice value ranges from -20 (fastest) to 20 (slowest). The nice value of a process (pid = 100) can be viewed by using the ps command as: ps -o pid,nice -p 100.
The nice value of a process can be changed by using the renice command. The renice command takes two arguments: the new priority and process ID. 


johndoe@laptop:~/Documents$ ps -o pid,nice -p 2292
PID NI
2292 0
johndoe@laptop:~/Documents$ renice -10 2292
2292: old priority 0, new priority -10
johndoe@laptop:~/Documents$
